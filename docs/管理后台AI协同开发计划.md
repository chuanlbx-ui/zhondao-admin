## 📋 项目概览

### 项目目标
- ✅ 完成中道商城管理后台的完整开发和集成
- ✅ 所有模块与真实后端API完整对接
- ✅ 达到80%以上的代码覆盖率
- ✅ 完整的文档和最佳实践沉淀

### 核心模块清单
```
后台系统
├── 1. 用户管理 (User Management)
│   ├── 用户列表、详情、创建、编辑、删除
│   ├── 用户等级体系
│   └── 权限和角色管理
│
├── 2. 商品管理 (Product Management)
│   ├── 商品列表、分类、标签
│   ├── 商品上下架、库存
│   └── 规格和属性管理
│
├── 3. 订单管理 (Order Management)
│   ├── 订单列表、详情、查询
│   ├── 订单状态转换
│   └── 订单统计和分析
│
├── 4. 店铺管理 (Shop Management)
│   ├── 店铺列表和审核
│   ├── 店铺升级申请处理
│   └── 店铺清算和财务
│
├── 5. 财务管理 (Finance Management)
│   ├── 通券管理（充值、转账、冻结）
│   ├── 佣金计算和发放
│   └── 收支统计和报表
│
└── 6. 系统管理 (System Management)
    ├── 菜单和权限配置
    ├── 参数配置管理
    └── 日志和审计
```

## 🔄 协同工作流规范

### 1. 每日工作流程
```
08:00 - 协调AI发布日报：昨日完成、今日计划、存在阻碍
09:00 - 开发AI开始工作，遇到问题立即沟通
12:00 - 中期进度检查（如需要）
17:00 - 代码自测和同行审查
18:00 - 提交代码和更新任务状态
```

### 2. 代码提交规范
```bash
# 分支命名规范
feature/TASK-001-user-list  # 新功能
bugfix/TASK-001-fix-xxx     # 问题修复

# 提交信息规范
feat(task-001): 实现用户列表查询功能
- 支持多条件搜索
- 分页加载
- 导出功能

# 关联任务
Relates to: TASK-001-1.1
Closes: #123
```

### 3. 代码审查标准
```typescript
// ✅ 必须通过的检查项：
□ 类型检查通过（无any）
□ 代码风格符合规范
□ 注释完整清晰
□ 测试覆盖率 ≥ 80%
□ 无性能问题
□ 无安全漏洞
□ API文档同步更新
□ 提交信息规范
```

### 4. 冲突解决流程
```
发现冲突
  ↓
AI自动检测并标记（git diff）
  ↓
协调AI评估冲突严重程度
  ↓
轻微（同一开发者修复）
中等（两个AI讨论合并方案）
严重（召集相关AI和架构师讨论）
  ↓
执行合并或协调
  ↓
验证和测试
  ↓
更新文档
```

## 📝 重要约定

### ✅ 必须做
1. **每日同步进度** - 更新任务状态和遇到的问题
2. **代码自测** - 提交前充分测试，不提交未测试代码
3. **写测试** - 关键业务逻辑必须有单元测试
4. **更新文档** - 代码修改同步更新文档
5. **遵守规范** - 严格遵守代码规范和提交规范
6. **及时沟通** - 遇到阻碍或冲突立即沟通

### ❌ 禁止
1. 不要提交未经自测的代码
2. 不要独立决策涉及多模块的改动
3. 不要忽视代码审查意见
4. 不要在没有测试的情况下合并代码
5. 不要修改他人未通知就修改的文件
6. 不要跳过文档更新

### 常见问题FAQ
Q: 如果遇到其他AI的模块有bug怎么办？

A: 充分讨论，最终由架构师AI决策；异议可上报协调AI

Q: 任务工时严重超期怎么办？
A: 立即向协调AI报告，分析原因，寻求帮助或调整计划
